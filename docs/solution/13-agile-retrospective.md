# Agile Retrospective

## Appris 

> Le TDD permet de mieux structurer le code, en se concentrant davantage sur les besoins métier (on ne s'amuse pas à faire du refactoring pour faire du refactoring). C'est pas mal.
> Nous avons appris le TDD, l'example mapping et le mob programming, nous avons surtout appris à structurer les idées et les règles métiers en amont afin de coder l'implémentation et les tests, en groupe plus facilement et rapidement


## Surpris

> L'étude des mutations : quand le code change, il faut revoir les tests. On plus l'habitude de ne rien toucher au tests tant qu'ils passent. Au final on passe plus de temps sur les tests que sur l'implémentation.
> La mise en place qui est d'effectuer tout l'example mapping, puis de faire du TDD, puis de faire du mob programming, paraît très lent mais une fois qu'on est lancé on remarque qu'il y a très peu besoin de revenir sur le code car on connaît les règles et les cas limites dés le début.
> Les tests de mutations sont très pratiques pour reconnaître les lacunes de nos tests et retrouver les bugs dans notre code

## Lundi

> Peut être pas faire des example mapping complets mais de structurer les règles métiers pour pouvoir coder plus facilement.
> Utiliser nottamenet lors du refactoring
> Utiliser des pipeline d'étude des mutations pour éviter les bugs
